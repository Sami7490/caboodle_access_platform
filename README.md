# Caboodle Access Platform

A full-stack clinical analytics and AI engineering portfolio project built from scratch — modeling a pediatric hospital's Access & Utilization subject area with a complete data pipeline, machine learning predictive models, a multi-tool LLM agent, and a locally-hosted Streamlit dashboard.

---

## What this project demonstrates

- **Data engineering**: PostgreSQL schema design, synthetic data generation, dbt Core transformation layer, Apache Airflow orchestration
- **Machine learning**: scikit-learn predictive models for no-show probability and 30-day readmission risk
- **AI engineering**: multi-tool Claude (Anthropic API) agentic loop, SQL guardrails, pgvector RAG, LLM observability, automated eval harness
- **Front end**: Streamlit dashboard with 7 sections covering clinical metrics, ML risk scores, semantic search, and a natural-language AI query assistant

## How to run locally

### Prerequisites
- PostgreSQL 18.4 with pgvector extension
- Python 3.13
- dbt Core 1.11.11 with dbt-postgres
- Apache Airflow 3.0.4
- Anthropic API key

### Setup

```bash
# 1. Create the database
createdb -U postgres caboodle_access

# 2. Build the raw schema
psql -U postgres -d caboodle_access -f sql/01_create_schema.sql

# 3. Install Python dependencies
pip3 install faker numpy psycopg2-binary scikit-learn pandas \
             anthropic sentence-transformers streamlit plotly

# 4. Generate synthetic data
python3 generator/generate_synthetic_data.py

# 5. Run dbt models
cd dbt/caboodle_access && dbt run && dbt test

# 6. Train predictive models
cd ../.. && python3 agent/train_noshow_model.py
python3 agent/train_readmission_model.py

# 7. Generate and embed clinical notes
python3 agent/generate_clinical_notes.py

# 8. Launch the dashboard
export ANTHROPIC_API_KEY="your-key-here"
streamlit run dashboard/app.py
```

### Dashboard sections
1. **Key Metrics** — no-show rate, readmission rate, patient days, appointment volume
2. **No-Show Analysis** — by department, appointment type, lead time
3. **Readmission Analysis** — by disposition, LOS distribution, monthly trend
4. **Patient Risk Lookup** — ML-generated no-show + readmission scores per patient
5. **Semantic Note Search** — pgvector similarity search over clinical notes
6. **AI Query Assistant** — natural-language queries backed by the full Claude agent
7. **LLM Observability** — live log of every Claude API call (tokens, latency, success)

## AI engineering highlights

**Multi-tool agentic loop**: The Claude agent doesn't just do single-shot NL-to-SQL. It decides which tool to call, calls it, sees the result, and loops until it has enough information to answer — including self-correcting when it generates invalid SQL by inspecting the schema and retrying.

**SQL guardrails**: Every SQL string generated by the agent passes through a validation layer that blocks all non-SELECT statements before they touch the database — regardless of what the user asks or how Claude interprets it.

**pgvector RAG**: 1,388 clinical notes are embedded locally using sentence-transformers (no API cost) and stored in PostgreSQL via pgvector. Semantic search finds clinically similar notes by meaning, not keyword matching.

**LLM observability**: Every Claude API call is logged to a Postgres table with prompt, response, token counts, latency, and success status — treating the LLM layer as a production system, not a black box.

**Eval harness**: An automated test suite runs 8 representative questions through the full agent and scores answers against expected results — providing a concrete, repeatable measure of agent reliability.
